概览

让 UTXO 也能安全承载状态机合约。

论文要解决的核心矛盾是：如何在不引入共享可变状态的前提下，提升 UTXO 模型对智能合约的表达能力，并让这种提升能被形式化证明。

  • 研究问题：账户模型靠全局状态换取表达力，但会显著增加并发语义复杂度；UTXO 能否兼顾表达力与语义简洁性？

First Principles

第一性原理卡片

只保留 3 张，并严格锚定论文里的真实设计与论证，不做脱离文本的泛化发挥。

为什么 EUTXO 能在不引入全局状态的情况下表达状态机？

论文把“状态”拆成了局部可携带的数据而不是全局可变存储。输出新增 datum 后，当前状态可随 UTXO 一起流动；验证器再通过 context 检查后继交易是否仍属于同一合约。于是状态被局部化到使用点，表达力上升，但共享可变状态并没有被重新引入。

核心原理：把状态变成被交易携带和验证的数据，而不是让所有节点长期维护一份全局可变内存。

为什么“有效时间区间”比直接暴露当前时间更合理？

作者非常在意验证阶段的确定性。若脚本能直接读取精确当前时间，它就可能因链上瞬时环境不同而分支，用户在提交前也无法准确模拟结果与资源消耗。把时间设计成交易自带的有效区间后，脚本只知道“当前 tick 落在区间内”，从而保留了链下可预测性。

核心原理：对脚本暴露“交易声明的时间边界”，而不是暴露“会实时变化的系统事实”。

为什么作者要证明 weak bisimulation，而不只说“能实现状态机”？

“能写出来”并不能保证链上行为与高层状态机语义一致。论文进一步要求：CEM 的每个相关转移都能落成合法交易，而链上与该合约相关的合法交易也能回译为 CEM 转移。同时又允许账本夹杂无关交易，因此采用的是弱双模拟而非强双模拟。

核心原理：开放账本上要证明的是“相关行为保持对应”，而不是“世界上每一步都只服务这一个合约”。

Abstract / 摘要

比特币和以太坊分别承载着当今最有价值、也最流行的两类加密货币，但它们采用的是两种相当不同的账本模型，分别称为 UTXO 模型与账户模型。与此同时，这两条公有链在所支持的智能合约表达能力上也有明显差异，而这并非偶然。以太坊之所以明确选择账户模型，就是为了更方便地支持表达能力更强的智能合约；另一方面，比特币选择 UTXO 也有其充分理由，其中包括：在复杂的并发与分布式计算环境中，它的语义模型仍保持简洁。

这就提出了一个问题：是否有可能在保留 UTXO 模型语义简洁性的同时，也支持高表达力的智能合约？本文对此给出肯定回答。我们提出扩展 UTXO（Extended UTXO, EUTXO），它是在比特币 UTXO 模型上的一种扩展，支持显著更强的验证脚本形式，包括能够实现通用状态机的脚本，以及能够在整条交易链上强制维护不变式的脚本。

为了展示该模型的能力，我们还提出了一种适合在账本上执行的状态机形式，它建立在 Mealy 机之上，被称为 Constraint Emitting Machines（CEM）。我们对 CEM 进行了形式化定义，展示了如何将其编译到 EUTXO 上，并证明了这两个系统之间的弱双模拟。本文全部工作都使用 Agda 证明助手完成了形式化。

关键词：Blockchain；UTXO；Functional programming；State machines

1 引言 Introduction

比特币是最广为人知、也最有价值的加密货币，它采用一种基于 UTXO（未花费交易输出）概念的图式账本模型。单笔交易由输入列表与输出列表组成，其中输出表示某个特定数量的加密货币，这些输出可以被后续交易的输入所花费。每个输出只能被恰好一个输入花费。此外，这些花费连接不允许形成环，因此一组互相花费的交易可以被视为一张有向无环图：若某笔交易有 m 个输入和 n 个输出，那么它在图中就是一个拥有 m 条入边和 n 条出边的节点。交易输入消耗的价值总和必须等于输出提供的价值总和，因此价值是守恒的。

某个输出是否能被输入消费，是由附着在该输出上的函数 ν 决定的，本文称其为输出的验证器。交易输入通过提供一个 redeemer 对象 ρ 来证明自己有资格花费该输出，使得 ν(ρ) = true；在比特币语境中，redeemer 常被称为 witness。最简单的情况里，redeemer 是对待花费交易哈希的私钥签名，验证器中嵌入对应公钥并完成校验。通过更复杂的验证器和 redeemer，也可以实现更复杂的协议。

这种基于图的数据流方式的好处在于，它与区块链的并发、分布式本性非常契合。特别是，它避免引入共享可变状态；而共享可变状态在并发与分布式计算中会带来极其复杂的语义问题。

尽管如此，UTXO 模型总体上，尤其是比特币，在可编程性表达能力方面长期受到批评。以太坊的账户式账本以及合约账户概念，很大程度上正是为了解决这些限制而提出的。不幸的是，它通过引入共享可变状态来实现这一点，从而显著增加了合约代码的语义复杂性。合约作者如果不理解这种语义的细节，就有可能引入安全问题，例如导致臭名昭著的 DAO 攻击的递归合约调用漏洞。

本文的核心贡献，是提出一种对基础 UTXO 账本模型的扩展，它一方面可被证明地提高表达能力，另一方面又保留 UTXO 图的数据流性质；尤其重要的是，它依然不引入任何共享可变状态。更具体地说，本文有以下贡献：

  • 提出 EUTXO 模型，并在第 2 节做非形式化说明，在第 3 节给出形式化定义。
  • 展示 EUTXO 能支持一种基础 UTXO 无法支持的状态机形式，即 Constraint Emitting Machines（CEM），相关内容见第 4 节。
  • 对 EUTXO 与 CEM 都给出形式化，并基于 Melkonian 等人的既有工作，在 Agda 证明助手中证明两者之间的弱双模拟。

第 5 节总结相关工作。EUTXO 模型将用于 Cardano 的账本层，而 Cardano 是 IOHK 正在开发的一条大型区块链系统。它也是 Cardano 智能合约平台 Plutus 的基础。Plutus 包含一门小型函数式编程语言 Plutus Core，用来实现脚本。虽然本文不打算展开 Cardano 的技术细节，但作者指出，读者可以在在线 playground 中体验 Plutus 平台。

作者还指出，未来工作包括对 EUTXO 与 Ethereum 账户模型做正式比较。

2 扩展 UTXO Extending UTXO

已有多种状态机形式被提议用来刻画那些超出基础 UTXO 模型能力范围的智能合约功能，例如一些工作就使用 Ethereum 的账户模型完成此事。然而，我们自然会追问：是否可以扩展基础 UTXO 模型，使其在不切换到账户模型的前提下，也支持更具表达力的状态机？

如果把一条连续交易链中的每一笔交易都看作状态机演化中的一个步骤，那么 UTXO 模型至少需要补上两项功能。第一，它必须能够维护状态机状态；第二，它必须能够强制整条交易序列始终沿用同一份合约代码，作者把这一点称为 contract continuity（合约连续性）。

为了维护状态机状态，我们把 UTXO 输出从原来的二元组 (ν, value) 扩展为三元组 (ν, value, δ)，其中 δ 是任意合约专属数据，即 datum。为了让验证器能够强制合约连续性，我们还把“尝试花费该输出的整笔交易”一起传给验证器执行。于是验证器就能检查当前消费交易产生的后继输出是否仍然使用同一合约的验证器代码，很多时候甚至就是同一个验证器。因此，当一笔交易 tx 中的某个输入携带 redeemer ρ 去花费输出 (ν, value, δ) 时，我们检查的是：

ν(value, δ, ρ, tx) = true

由于 δ 可以携带任意数据，而验证器 ν 又能对消费它的交易 tx 施加任意有效性约束，因此 EUTXO 的表达力其实超出了“支持状态机”这一范围。不过本文把讨论收束在状态机实现上，把更广义的计算模式留给未来研究。

作为简单示例，作者讨论一个 n-of-m 多重签名合约。具体来说，系统中有一笔价值为 value_msc 的资产，它必须获得一个预先固定的 m 人所有者集合中至少 n 人的批准才能被花费。在 plain UTXO（例如比特币）中，多签方案通常要求所有签名在线下先收集完；而在 Ethereum 与 EUTXO 中，签名收集过程本身可以在链上完成，从而无需额外的链下通信。

为此，作者使用图 1 所示的状态机，并假定阈值 n 以及授权签名集合 sigs_auth 已写入合约代码。在 EUTXO 的实现里，作者使用验证器 ν_msc 和 datum δ_msc 来锁定 value_msc。datum 存放机器状态：若只是持有被锁定资产，则状态为 Holding；若正在为某笔支付收集签名，则状态为 Collecting((value, κ, d), sigs)。初始输出为 (ν_msc, value_msc, Holding)。

验证器 ν_msc 通过查看花费输入上的 redeemer 来判断应走哪条状态转移边。该 redeemer，也就是状态机输入，可以是四种形式之一：Propose(value, κ, d) 表示提出一笔支付；Add(sig) 表示为该支付追加一个签名；Cancel 表示在截止时间过后取消提案；Pay 表示在收集到足够签名后执行支付。随后，验证器会检查消费交易 tx 是否是新机器状态的合法表示。这意味着 tx 必须继续用 ν_msc 锁定相应价值，并且新的 datum δ'_msc 必须是按照状态图从 δ_msc 演化得到的后继状态。

EUTXO 的表达能力远不止这个链上多签例子。作者进一步指出，金融合约领域特定语言 Marlowe 的完整功能，也已经成功实现为运行在 EUTXO 模型上的状态机。

3 形式模型 Formal Model

这一节给出 EUTXO 的形式化定义。作者采用的是直接的集合论风格表达，它有两个目的：第一，这种写法几乎可以直接翻译成 Haskell 之类的实现语言；第二，它很适合进一步做机械化形式化。后文第 4 节会基于这一点继续推进。

作者说明，这里给出的定义本质上仍然建立在 Zahnentferner 对带脚本 UTXO 加密货币的抽象模型之上，只是加入了前面介绍的三项增强：datum、context 和有效时间区间。

3.2 EUTXO：增强型脚本 EUTXO: Enhanced Scripting

相较于标准 UTXO 模型，作者做出的第一项改变是：交易输出除了验证器外，还允许携带一段名为 datum 的数据，并在验证时作为额外参数传给脚本。这样一来，合约就能在不改变自身“代码”的情况下携带某些状态，而作者正是用这一点来承载后文的状态机状态。

第二项改变是：验证器在执行时还能收到一个类型为 Context 的额外参数，也就是当前正在验证的交易的某种摘要信息。正是这个摘要，使验证器能够施加强于基础 UTXO 的条件，尤其是检查当前交易输出，从而保证合约连续性。

第三项改变是：作者通过为交易加入有效时间区间，给脚本提供了受控的时间访问能力。该区间表示交易在哪些 tick 期间允许被处理，它是对“time-to-live”的一种推广。验证阶段的脚本只知道当前 tick 位于此区间中，但不知道其精确值。

最后，验证器收到的所有参数，也就是 redeemer、datum 与 Context，都会统一编码为 Data 类型。因此客户端有责任把自己想使用的具体类型编码成 Data，并在验证器内部再解码回来。

3.3 EUTXO 模型的形式化描述 A Formal Description of the EUTXO Model

这一节正式给出 EUTXO 模型的类型与操作。图 3 列出了其所需的类型和运算，其中一部分在论文中定义，另一部分由底层账本提供，作者称之为 ledger primitives。

作者沿用比特币的术语，把交易输出的目标称为“地址”。在本文系统里，它只代表脚本地址，也就是脚本的哈希；而在更完整的系统中，地址还可能包含公钥地址等其他形式。tick 则是账本系统中单调递增的推进单位，可类比为多数区块链里的区块号或区块高度。作者假定每个账本状态都存在某种“当前 tick”。

交易输入在模型中被表示为集合，而输出被表示为列表。作者特别说明，输出不能简单建模为集合，因为我们需要一种方式唯一标识某个输出，以便后续输入引用它，而“交易 id + 输出索引”的组合天然更适合列表；此外，如果采用结构相等性，那么两个“向同一地址支付同一数额”的输出就会被视作相等，这不符合现实中输出必须具有独立身份的要求。

验证器脚本与完整 datum 对象虽然概念上属于被花费的输出，但在实现上是由交易输入提供的；输出本身只保存它们的地址或哈希。这样做的主要动机是节省内存，因为 UTXO 集在验证期间必须被快速访问并驻留内存，因此输出尽量保持常数大小会更有利，而较大的脚本与 datum 只在真正消费该输出时才提交。

作者也承认，这会带来一个问题：花费者如何知道自己应提供哪一份验证器和 datum 才能匹配输出上的哈希？这可以通过链下机制解决，也可以考虑未来加入链上发现机制；Cardano 将提供某种相关机制，但本文不对其展开。此外，交易费用、forge 字段以及其他附加元数据，在现实系统中当然存在，但它们都与本文主线无关，因此被省略。账本结构本身也被简化成交易列表，而不是现实世界中的完整区块链结构；作者真正依赖的，只是交易能够被唯一索引和检索。

3.4 Context 类型 The Context Type

回顾第 3.2 节：当某个交易输入正在被验证时，验证器会收到一个类型为 Context 的对象，它以 Data 编码，包含与当前交易相关的信息。图 4 定义了这个类型以及一些相关类型。

从设计上看，Context 是图 3 中 Tx 类型信息在“某个特定输入接受验证”这一语境下的摘要版，并被整理成脚本容易消费的形式。作者列出三点变化。第一，InputInfo 会附加来自被花费输出的信息，特别是该输出附带的价值。第二，Context 中包含一个索引，标识当前正在验证的是哪个输入，这样脚本就能识别自己的地址。第三，验证器在 Context 中不再以脚本本体出现，而是只以地址出现，从而方便比较相等性，而无须要求脚本语言能够表示自身程序。

作者假设存在如下函数：

toContext : Tx × Input × Ledger → Context

它把一笔交易、其中某个输入以及当前账本状态，总结成验证器可消费的上下文对象。

这一设计最重要的性质是确定性。Context 中提供的信息完全由交易自身决定，因此验证期间的脚本执行也是完全确定的。用户在提交交易之前，就可以准确模拟脚本是否通过验证以及资源消耗是多少；这对于按脚本执行收费的系统尤其重要。作者指出，许多系统破坏这种性质的方式，是允许脚本直接读取某个会变化的全局量，例如当前时间。在本文系统中，这个角色由当前 tick 扮演，而作者正是通过“只暴露有效时间区间而非精确 tick”来避开这种不确定性。

3.5 EUTXO 交易的有效性 Validity of EUTXO Transactions

图 6 定义了在当前 tick 下，一笔交易 t 对于某个有效账本 l 何时有效。作者把 Zahnentferner 文中的定义 6 与定义 14 结合起来，主要差异体现在规则 6。一个账本是有效的，当且仅当它要么为空，要么可写成 t :: l' 的形式，其中 l' 本身有效，并且 t 对于 l' 有效。

现实世界中的区块链往往还有创造或销毁价值的特殊规则，它们通常很具体，也与本文主题无关。因此作者只在规则 4 中放入一个简单的创世条件，允许账本中的第一笔交易创建价值。

另一个看似潜在的问题是：函数 getSpentOutput 会调用 lookupTx，而后者自然可能查找失败。不过规则 3 保证：在验证过程中，所有交易输入都引用真实存在且尚未花费的输出。在这一前提下，对于本文关心的交易，lookupTx 总会成功。

4 EUTXO 的表达能力 Expressiveness of EUTXO

这一节里，作者引入了一类状态机，用来直接而自然地建模运行在 EUTXO 账本上的智能合约。作者选择的这类机器与 Mealy 机非常接近。Mealy 机的转移函数不仅返回新状态，还会返回一个输出值；在本文里，这个输出值被解释成一组施加在当前交易上的约束。

作者并不宣称这类状态机已经完全刻画了账本的全部能力；之所以选择它，是因为它形式简洁，却足以覆盖相当广泛的智能合约使用场景。作者还说明，相关结果已在 Agda 中完成机械化，建立在第 3 节模型的可执行规格之上。

4.1 约束发射机 Constraint Emitting Machines

作者提出 Constraint Emitting Machines（CEM），它建立在 Mealy 机之上。一个 CEM 由状态类型 S、输入类型 I、一个判断某状态是否为终态的谓词 final : S → Bool，以及一个给出所有合法转移的函数组成：

step : S → I → Maybe (S × TxConstraints)

这个函数从源状态和输入符号出发，返回目标状态与一组交易约束；若不存在合法转移，则返回 Nothing。

作者强调，他们研究的这类状态机与教科书中的典型 Mealy 机有四点不同。第一，状态集合可以是无限的。第二，它没有“初始态”概念，因为链上层面无法直接强制合约必须从某个特定初始态起步。因此，每个合约都需要首先建立某种初始信任来启动过程。作者提到，未来可以在当前单状态关系之上建立 trace simulation，从而保证只有从合法初始态开始的轨迹会出现在链上，并把状态机的归纳性质迁移到账本层。

第三，终态的含义被重新定义了。传统上终态意味着“机器可以在此停止”，但如果在链上也允许这样，就会让陈旧状态残留在 UTXO 集中。因此在 CEM 中，终态意味着“机器必须停止”：从这一点往后不会再有继续转移，而终态也不会出现在 UTXO 集中。作者把这种含义与 stopped process 联系起来。

第四，输出值的集合被固定为“约束”，也就是对实现该状态转移的交易施加结构性要求。作者承认，目前形式化里考虑的只是有限的一阶约束，但这一设计以后可以在不大幅改动证明结构的前提下继续扩展。

4.2 把转移实现为交易 Transitions-as-Transactions

作者希望把一个由 CEM 定义的智能合约 C 编译成真正运行在链上的智能合约。基本想法是：从状态机的 step 函数派生出验证器脚本，用 datum 保存机器状态，用 redeemer 提供转移信号。于是，CEM 中的一个合法转移就会对应链上的一笔合法交易。

更具体地说，该验证器要确保三件事：当前交易确实把状态推进到了合法目标状态；当前交易满足状态机发射出的约束；如果目标状态是终态，那么交易就不应再创建继续该状态机的输出。作者将其写作：

validatorC(s, i, txInfo) = true if s --i→ (s', tx≡) and satisfies(txInfo, tx≡) and checkOutputs(s', txInfo) false otherwise

需要注意的是，状态机里的 step 函数返回新状态，而链上的验证器只返回布尔值。在链上，下一状态是通过继续该状态机的交易输出携带出来的，验证器只负责确认“你给出的下一状态是否正确”。用户完全可以在链下先运行一步 step，从而算出下一状态应该是什么。

4.3 行为等价性 Behavioural Equivalence

既然已经说明如何把状态机编译为链上合约，接下来的问题就是：我们如何确信这个链上合约的行为确实符合原始状态机语义？作者希望证明两件事。第一，CEM 中任何合法转移都能对应到链上的合法交易；第二，链上任何与该合约相关的合法交易，也都能对应回某个合法 CEM 转移。作者把这两条性质分别称为 soundness 与 completeness。

虽然状态机也是自动机，但自动机理论里的 trace equivalence 对这里来说过于粗糙。作者希望把状态机和账本都看成正在运行的进程，因此转而使用并发理论中的 bisimulation。不过，如果采用强双模拟，就要求账本上的每一步都与状态机中的一步一一对应；现实账本上会发生大量与当前合约无关的交易，因此作者采用的是弱双模拟：账本可以有额外的内部步，只要与目标合约相关的行为仍然保持对应即可。

作者首先定义过程关系：只要某个 CEM 状态 s 出现在当前 UTXO 集中，并被从该 CEM 派生出的验证器锁定，就记账本 l 与状态 s 对应：

l ∼ s

随后定义账本步。作者只考虑有效账本，因此一次有效账本步就是把一笔相对于当前账本 l 有效的新交易 tx 提交到账本上，得到扩展后的新账本 l'：

l --tx→ l'

命题 1 给出 soundness：给定一个合法的 CEM 转移 s --i→ (s', tx≡)，以及与源状态 s 对应的有效账本 l，我们总能构造出一笔合法交易提交，使得扩展后的账本 l' 与目标状态 s' 对应。作者额外说明，这里还要求被省略的那些约束在当前账本中可满足，并且目标状态不是终态，因为终态不会在账本里留下继续输出作为见证。如果扩展 correspondence 定义，也可以把终态纳入，但为保持简单，本文没有这样做。

命题 2 给出 completeness：如果账本中发生一次合法转移 l --tx→ l'，并且某个 CEM 状态 s 与原账本 l 对应，那么要么这笔交易与当前 CEM 无关，此时新账本 l' 仍对应原状态 s；要么这笔交易与该 CEM 有关，那么就能展示出相应的输入、目标状态和约束，使之构成一个合法 CEM 转移。同样地，若目标状态是终态，则存在与上面相同的对应性边界。

soundness 与 completeness 共同给出了本文所需的弱双模拟。作者最后提醒，这里的双模拟概念与教科书上的标准定义略有差异，因为它还带有关于约束以及终态特殊处理的额外前提。

5 相关工作 Related Work

Bitcoin Covenants 允许比特币交易限制未来输出的使用方式，包括把某种约束传播到后续输出。这为合约连续性和简单状态机实现提供了能力。作者承认，他们的工作受到了 Covenants 的启发，但指出本文引入 datum 是新的贡献，并且使状态传递更为简洁。

Bitcoin Modelling Language（BitML）是一种面向运行在比特币上的智能合约的理想化过程演算。BitML 合约的语义本质上就是一个带标签的状态转移系统，也就是状态机。不过，由于 plain UTXO 没有本文这些扩展，它的构造远没有这么直接，通常需要相当多的链下通信来搭建整个过程。更重要的是，BitML 的编译方案只涉及受限形式的状态机，而本文处理的是更一般的状态机，允许用户自定义状态类型和输入类型。BitML 建立在作者同一批人较早提出的比特币交易抽象模型之上；而本文的重要贡献之一，正是把那种抽象模型扩展到可以覆盖 Cardano 新增的功能。

Ethereum 及其智能合约语言 Solidity 当然足够强大，可以实现状态机，因为它们原生支持全局合约实例与状态。但这种方法有明显缺点：合约状态是全局的，并且必须由所有核心节点长期保存。在 EUTXO 模型中，合约状态被局部化到真正使用它的地方，而由客户端承担管理它的责任。

Scilla 是一种把智能合约写成状态机的中间层语言，它会编译到 Solidity，并且适合做形式化验证。由于 Scilla 支持 Ethereum 的异步消息机制，Scilla 合约对应的是一种更丰富的自动机类，即 Communicating State Transition Systems。作者表示，未来计划把这类状态机与本文的 CEM 做正式比较，这也会为系统比较 Ethereum 的账户模型与 Cardano 的 UTXO 模型铺路。

最后，还有工作尝试使用 timed automata 建模比特币合约，并借助 UPPAAL 模型检查器半自动验证具体合约的时间性质。虽然这是一种务实的验证方式，但它并未形式化地宣称这种自动机类别与比特币合约的真实语义严格对应。相比之下，本文的双模拟证明试图跨越的正是这道语义鸿沟。
